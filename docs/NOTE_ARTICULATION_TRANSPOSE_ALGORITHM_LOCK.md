# 音符、奏法与移调算法锁定规范

版本：`NAT-1.0`  
锁定日期：2026-07-10  
适用范围：本项目后续所有“音符修复 / MIDI 优化 / 奏法处理 / 升降调”工作。

## 1. 锁定目标

算法必须先保证 BDO 能正确发声，再保证音乐结构不被破坏，最后才做表现性优化。固定优先级如下：

1. 数据与导出合法；
2. 原旋律、和声、节奏和人工演奏信息不被意外改变；
3. 音符落入目标 BDO 乐器的实测可触发键位；
4. 同拍奏法合法且与乐器匹配；
5. 在证据充分时改善力度、时值、连断与奏法；
6. 风格只改变建议权重，不能凌驾于以上硬约束。

未经新的游戏内 A/B 验证和测试用例，不得放宽本规范的硬约束。算法升级使用 `NAT-1.x`；改变优先级、允许自动改音高或放宽奏法写入条件时必须升为 `NAT-2.0`。

## 2. 覆盖的项目乐器

以仓库当前 26 种 GUI 乐器为完整覆盖集，不把内部/旧资源 `0x03`、`0x09` 擅自加入 GUI：

| 家族 | 乐器 ID |
| --- | --- |
| 弦乐 / 吉他 / 贝斯 | `00, 06, 08, 0A, 0E, 0F, 10, 12, 24, 25, 26` |
| 木管 / 铜管 | `01, 02, 0B, 27, 28` |
| 键盘 / 合成器 | `07, 11, 14, 18, 1C, 20` |

具体名称、游戏音域、奏法编号和采样键位继续以 `README.md`、`pyside_bdo_gui.py`、游戏 `list*` 反解数据及 Wwise 映射为项目内事实来源。真实乐器资料用于判断“是否自然”，不能覆盖游戏实测音域。

## 3. 不可变规则

### 3.1 音符

- MIDI 音高只按整数键号 `0..127` 处理；中央 C 固定为 MIDI 60。界面可采用不同八度命名，但内部键号不能随命名变化。
- 每个 Note On 必须有对应的 Note Off；修重叠时不得制造悬挂音。
- 默认保留原音高、音程、和声、起音、时值、力度、已有 `ntype`、CC 与人工微时差。
- 去重只删除同轨、同起点容差、同音高、同奏法的真实重复事件。
- 不用逐音 `clamp` 修复越界音，因为那会改变音程与和声。
- 打击乐键号代表鼓件/手势，不参与旋律升降调；只允许走已验证的 GM→BDO 打击映射。
- 电吉他 `FX(C2~G2)` 是触发区，不是普通旋律音区；除非轨道明确标记为 FX，自动算法不得生成、搬移或吞并这些触发音。

### 3.2 奏法

- 先按起点容差 12 ms 建立“同拍音组”，再决定奏法，禁止逐音贪心后产生同拍冲突。
- BDO 同一节拍只能保留一个奏法；同拍候选冲突时按“人工已有 > 游戏 A/B 已验证 > 置信度 > 基本奏法”裁决，无法安全裁决则整组保持原状。
- 已有合法音符级或轨道级奏法永远优先，除非用户明确要求替换。
- 未通过游戏内 A/B 的 `乐器 × ntype` 只能显示建议，不能自动写入。
- 奏法必须同时满足：目标乐器支持、音区合适、织体合适、非同拍冲突、非 FX 触发误判、置信度达标。
- 和弦、交叉声部、句尾长音、调性不稳定材料采用保守策略；滑音和颤音不得跨声部猜测。

### 3.3 移调与音高修复

必须区分三种操作，界面和报告不得混称：

1. **调性移调**：用户要求升/降 `s` 个半音，所有非打击声部统一 `pitch += s`，保持音程与和声关系。
2. **八度适配**：为适配乐器音域，以 `12k` 半音移动单个完整声部；优先保持音色角色和声部排列。
3. **异常音修复**：只处理可证明的数据错误，例如越界键号、重复事件、悬挂音或错误鼓映射；不得把“听起来奇怪”直接改成邻近音。

对每条旋律轨，设原最低/最高音为 `p_min/p_max`，目标游戏音域为 `L/U`，合法统一移调区间固定为：

```text
L - p_min <= s <= U - p_max
```

- 若用户指定的 `s` 在区间内，直接使用。
- 若指定 `s` 越界，先报告冲突；自动适配只能在用户允许时追加八度 `12k`。
- 若区间为空，禁止压缩或逐音折返；建议拆轨、换乐器或人工改写。
- 全曲统一移调优先于逐轨移调；逐轨八度适配不得造成低音越过旋律、内声部越位等明显声部交叉。
- 任何移调后必须重新运行：音域检查、采样键位检查、FX 区检查、同拍奏法检查。

## 4. 固定处理流水线

```text
输入快照
  → 结构校验（事件、范围、Note On/Off、排序）
  → 轨道/声部识别（旋律、低音、和声、节奏、打击、FX）
  → 乐理上下文（拍号、速度、强弱拍、乐句、织体、调性置信度）
  → 音高方案（用户调性移调 → 可选八度适配 → 音域/采样复检）
  → 时值与重叠修复
  → 力度微调
  → 奏法候选生成
  → 同拍冲突消解与证据门控
  → 只读预览/差异报告
  → 用户确认后写入
  → 导出前完整复检
```

音高方案必须先于奏法判断，因为移调会改变常用音区、FX 触发区和采样键位。结构修复先于力度和奏法，避免对随后会被删除的重复音做表现处理。

## 5. 奏法候选的统一评分

所有乐器共用可解释评分，不允许为某一首歌写不可追踪的特例：

```text
score = technique_match
      + phrase_match
      + meter_match
      + texture_match
      + register_match
      + style_prior
      - same_onset_conflict
      - voice_crossing_risk
      - cadence_risk
      - unverified_mapping_risk
```

- `unverified_mapping_risk` 是硬门控：未 A/B 验证时，无论总分多高都只能建议。
- `style_prior` 是小权重软特征，不能单独触发奏法。
- 同一乐句最多给 3 个自动候选，候选之间至少间隔 2 个音符；具体阈值沿用 `OptimizerConfig`，修改必须有回归测试。

## 6. 音乐类型的处理方式

不建立“列完所有曲风”的封闭枚举。音乐类型采用可叠加标签，覆盖古典/管弦、室内/独奏、流行、摇滚/金属、爵士/布鲁斯、放克、电子/合成、氛围、民谣/世界音乐、舞曲、进行曲、影视/游戏配乐和实验音乐，也允许混合类型。

风格层只提供以下软先验：

- 节奏密度与可接受量化强度；
- 强弱拍与切分倾向；
- 奏法稀疏度；
- 乐器角色倾向（主旋律、低音、和声、节奏、纹理）；
- 力度曲线与句法密度。

当风格未知、混合或置信度低时，权重退回中性；不得因为“像某种风格”而改音高、和弦或生成新音符。

## 7. 乐器家族规则

- **弓弦**：连续旋律默认保持；颤音必须有邻音往返证据；拨弦/跳弓等无可靠游戏映射时只建议。
- **吉他/贝斯**：闷音要求短、分离且为重复 riff；slap 要求高力度短音；泛音只作高音区稀疏候选；FX 不自动生成。
- **竖琴**：滑奏要求同向快速连续级进；普通音阶不等于滑奏；和弦奏法以同拍整组判断。
- **木管/铜管**：按单声部和换气乐句判断；断奏要同时有短时值和可听间隔；长音力度层不能把重音误缩成短音。
- **钢琴**：踏板依赖 CC64 或跨后续和声的保持证据；不把所有长音变踏板。
- **合成器**：可用长音、纹理、力度做音色建议，但未知的玛勒尼恩映射一律不自动写入。
- **打击乐/手碟/钹/手鼓**：节奏与力度优先；有音高的手碟只使用实测键位；无固定音高乐器不运行调性算法。

## 8. 输出与可追溯性

每次优化报告至少记录：

- 算法版本 `NAT-1.0`；
- 原值、建议值、最终值；
- 乐器、声部、音域和所用移调量；
- 每个奏法候选的规则、置信度、证据级别和未应用原因；
- 被保护的人工事件；
- 仍需拆轨、换乐器或游戏 A/B 的项目。

任何自动写入都必须可撤销。预览与实际应用必须使用同一份不可变候选结果，不能在用户确认后重新随机分析。

## 9. 验收门槛

- 26 种 GUI 乐器逐一有音域/采样覆盖测试。
- 移调覆盖：正负半音、正负八度、无合法区间、打击轨、FX 轨、混合轨、声部交叉。
- 奏法覆盖：同拍冲突、人工奏法保护、未验证映射只建议、和弦屏障、邻音颤音、滑音方向、句尾保守。
- 优化前后除明确批准字段外，音符数量、音程关系、和声集合和事件顺序可解释。
- 导出前不存在越界键号、无效 `ntype`、错误鼓映射或普通旋律误入 FX 触发区。
